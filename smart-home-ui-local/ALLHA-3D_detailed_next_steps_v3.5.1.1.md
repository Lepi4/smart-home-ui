# 1. Следующий рекомендуемый шаг: v3.5.2 — Upload общего плана + reset image

## Цель

Пользователь должен загрузить, заменить и сбросить общий план через UI, без ручной замены файлов в Docker.

## UI

Добавить в настройки раздел:

```text
Картинки / План

Общий план
[Загрузить общий план]
[Заменить общий план]
[Сбросить картинку]
```

## API

Добавить:

```text
POST /api/images/overview
DELETE /api/images/overview
```

## Поведение upload

```text
1. Пользователь выбирает файл.
2. Сервер проверяет формат и размер.
3. Сервер сохраняет оригинал в /data/images/originals/.
4. Сервер создаёт рабочую версию overview.webp в /data/images/overview/.
5. Сервер обновляет images_meta.json.
6. UI перезагружает общий план.
```

## Поведение reset

```text
1. Пользователь нажимает “Сбросить картинку”.
2. UI показывает подтверждение.
3. Сервер создаёт backup текущей картинки и images_meta.json.
4. Сервер удаляет /data/images/overview/overview.webp.
5. images_meta.json получает overview: null.
6. UI начинает использовать fallback-картинку.
```

## Важное предупреждение

Если новая картинка имеет другое соотношение сторон, показывать:

```text
Новая картинка имеет другое соотношение сторон.
Маркеры и зоны могут визуально сместиться.
Перед заменой будет создан backup.
```

## Backup

Перед заменой или сбросом общего плана создать backup:

```text
/data/backups/images-overview-before-replace-YYYYMMDD-HHMMSS/
/data/backups/images-overview-before-reset-YYYYMMDD-HHMMSS/
```

Минимум сохранить:

```text
текущую рабочую картинку
images_meta.json
layout.json
```

## Критерии готовности v3.5.2

```text
- общий план можно загрузить через UI;
- общий план сохраняется в /data/images/overview/overview.webp;
- оригинал сохраняется в /data/images/originals/;
- /api/images показывает custom overview;
- после перезапуска add-on картинка остаётся;
- общий план можно сбросить к fallback;
- перед заменой/сбросом создаётся backup;
- README и FAQ обновлены;
- новый md-план создан вместе с архивом.
```

# 3. Базовые правила v3.5.x

## 3.1. Runtime-данные только в /data

Пользовательские данные нельзя хранить внутри Docker image.

Все данные пользователя должны быть здесь:

```text
/data/layout.json
/data/addon_config.json
/data/source_config.json
/data/ui_state.json
/data/devices.json
/data/devices.js
/data/lovelace-source.js
/data/lovelace_raw.json
/data/attention_rules.json
/data/security_rules.json
/data/rooms.json
/data/images/
/data/images/overview/
/data/images/rooms/
/data/images/originals/
/data/images/images_meta.json
/data/backups/
```

## 3.2. public/ только fallback/demo

В Docker image можно хранить:

```text
public/assets/default-overview.png
public/assets/default-room.png
public/brand-logo.svg
public/favicon.png
```

Но пользовательские картинки:

```text
НЕ public/assets/user-room.png
НЕ hardcode в Docker
НЕ ручная замена файлов
```

Только `/data/images`.

## 3.3. Комнаты не создаём вручную

Комнаты берутся из:

```text
1. Lovelace card / section title
2. имя устройства или entity_id
3. Home Assistant Area
4. Неразмещённые
```

Запрещённые действия:

```text
создать комнату вручную
удалить комнату вручную
переименовать комнату как источник истины
```

Разрешённые действия:

```text
назначить картинку найденной комнате
скрыть/показать найденную комнату
изменить порядок отображения
alias/отображаемое имя, если нужно
```

# 5. v3.5.2 — Upload общего плана

## Цель

Пользователь может загрузить свой общий план квартиры/дома через UI.

## UI

В настройках добавить раздел:

```text
Картинки / План
```

Внутри:

```text
Общий план
[Загрузить общий план]
[Заменить общий план]
[Вернуться к fallback]
```

## API

Добавить:

```text
POST /api/images/overview
DELETE /api/images/overview
```

## Поведение upload

1. Пользователь выбирает файл.
2. Сервер проверяет формат.
3. Сервер сохраняет оригинал.
4. Сервер создаёт рабочую версию.
5. Обновляет `images_meta.json`.
6. UI перезагружает карту.

## Хранение

Оригинал:

```text
/data/images/originals/overview-original.<ext>
```

Рабочая версия:

```text
/data/images/overview/overview.webp
```

Metadata:

```json
{
  "overview": {
    "src": "/media/images/overview.webp",
    "originalFilename": "my-plan.png",
    "originalWidth": 5013,
    "originalHeight": 2823,
    "processedWidth": 2500,
    "processedHeight": 1408,
    "format": "webp",
    "aspectRatio": 1.776,
    "updatedAt": "2026-05-07T00:00:00.000Z"
  }
}
```

## Backup

Перед заменой общего плана:

```text
создать backup текущей картинки
создать backup images_meta.json
создать backup layout.json
```

## Предупреждение

Если новая картинка имеет другой aspect ratio:

```text
Новая картинка имеет другое соотношение сторон.
Маркеры и зоны могут визуально сместиться.
Перед заменой будет создан backup.
```

## Критерии готовности

```text
- общий план можно загрузить
- общий план сохраняется в /data
- после перезапуска add-on картинка остаётся
- можно вернуться к fallback
- backup создаётся
```

# 12. v3.5.4 — Kiosk Tile Mode / второй экран плиток

## Цель

Добавить второй kiosk-экран: **плитки вместо карты**.

Это не отдельный редактор и не отдельная структура дома.  
Плиточный режим должен строиться из уже существующей карты, комнат, layout и HA states.

## Где доступен

Только в kiosk mode.

Переключатель:

```text
Карта / Плитки
```

Последний выбранный режим запоминается локально для конкретного устройства:

```text
localStorage:
kioskView = map | tiles
```

Это важно, потому что:

```text
настенная панель может открываться в плитках
ПК может открываться на карте
мобильный может помнить свой последний режим
```

---

## Главная логика плиточного режима

Комнаты в плиточном режиме должны быть **те же самые**, что и на карте:

```text
Общий план
Гостиная
Кухня
Спальня
Санузел
...
```

Источники комнат не меняются:

```text
1. Lovelace card / section title
2. имя устройства или entity_id
3. Home Assistant Area
4. Неразмещённые
```

Плитки создаются только из элементов, которые уже нанесены на карту:

```text
устройства на общем плане
устройства на картинках комнат
системные датчики
важные badges/metrics, если они размещены
```

Не показывать в плитках:

```text
неразмещённые устройства
скрытые устройства
удалённые устройства
устройства без координат
устройства, которых нет на карте/комнате
```

---

## Структура плиточного экрана

### Если открыт “Общий план”

Показывать все размещённые элементы, сгруппированные по комнатам:

```text
Общий план
  [Погода] [Общие датчики] [Системные элементы]

Гостиная
  [Свет] [Кондиционер] [Шторы] [Температура]

Кухня
  [Свет] [Датчик протечки] [Температура]

Спальня
  [Шторы] [Свет] [Климат]
```

### Если открыта конкретная комната

Показывать:

```text
заголовок комнаты
плитки только элементов этой комнаты
```

Пример:

```text
Гостиная
[Свет точки] [Люстра остров] [Кондиционер] [Температура] [Влажность]
```

### Заголовки групп

Каждая группа должна иметь заголовок:

```text
Гостиная
Кухня
Спальня левая
Общий план
```

Заголовки компактные, но читаемые.

---

## Автомасштаб плиток

Главное требование:

```text
плитки должны автоматически масштабироваться так, чтобы всё уместилось на экране
```

Подход:

```text
1. Посчитать количество групп и плиток.
2. Определить доступную ширину/высоту kiosk viewport.
3. Подобрать размер плитки.
4. Подобрать количество колонок.
5. Уменьшить шрифт/иконки в разумных пределах.
6. Не допустить горизонтального скролла.
```

Рекомендуемые ограничения:

```text
минимальная плитка: 90×60 px
обычная плитка: 130×80 px
крупная плитка: 170×100 px
```

Если всё равно не помещается:

```text
перейти к компактному режиму
или включить страницы
```

---

## Возможные страницы плиток

Пока не делать обязательно, но заложить в план.

Если устройств слишком много, можно добавить режим страниц:

```text
Страница 1: Общий план + Гостиная

```

Или:

```text

```

Возможное место настройки:

```text

```

Пока решение не фиксируем окончательно.

Нужно рассмотреть варианты:

```text
1. автомасштаб без страниц
2. страницы по комнатам
3. ручная настройка страниц
4. автостраницы по количеству плиток
```

Первый этап лучше сделать без ручного меню страниц:

```text
v3.5.4: автомасштаб
позже: страницы, если экран перегружен
```


Плиточный режим только отображает уже размещённые элементы.

## A. Удалить / сбросить пользовательскую картинку

### Проблема

В плане есть загрузка и замена картинок overview/rooms, но нужна отдельная функция:

```text
Удалить пользовательскую картинку / Сбросить к fallback
```

Иначе пользователь сможет загрузить или заменить картинку, но не сможет удобно вернуться к встроенной fallback-картинке.

### Где нужно

Для общего плана:

```text
Настройки → Картинки / План → Общий план
[Загрузить]
[Заменить]
[Сбросить картинку]
```

Для комнат:

```text
Настройки → Картинки комнат → Комната
[Загрузить]
[Заменить]
[Сбросить картинку]
```

### Поведение

Если пользователь нажимает “Сбросить картинку”:

```text
1. Показать подтверждение.
2. Создать backup текущей картинки и images_meta.json.
3. Удалить пользовательскую рабочую картинку из /data/images.
4. Оставить оригинал в /data/images/originals/ или перенести в backup.
5. Обновить images_meta.json.
6. UI начинает использовать fallback-картинку из public/assets.
```

### Важно

Сброс картинки не должен удалять:

```text
layout
markers
zones
devices
rooms settings
```

Но нужно показывать предупреждение:

```text
Картинка будет сброшена к fallback.
Layout и маркеры сохранятся, но могут визуально не совпадать с fallback-планом.
```

### API

```text
DELETE /api/images/overview
DELETE /api/images/rooms/:room_id
```

### Критерии готовности

```text
- общий план можно сбросить к fallback;
- картинку комнаты можно сбросить к fallback;
- backup создаётся;
- layout не удаляется;
- UI сразу обновляет картинку.
```

## B. Очистить маркеры и очистить зоны

### Проблема

Нужны быстрые сервисные функции для переинициализации layout без полного удаления проекта:

```text
Очистить маркеры
Очистить зоны
```

### Где разместить

Только admin mode:

```text
Настройки → Layout / Обслуживание
```

или:

```text
Настройки → Информация / диагностика → Layout tools
```

### Очистить маркеры

Функция удаляет размещение устройств с карт/комнат.

Что очищать:

```text
marker positions
device placement on overview/rooms
custom marker layout
```

Что не трогать:

```text
devices list
Lovelace import data
rooms
zones
images
security rules
attention rules
PIN
dangerous overrides
```

После очистки устройства должны попасть в:

```text
Неразмещённые устройства
```

### Очистить зоны

Функция удаляет зоны комнат с overview.

Что очищать:

```text
overview zones
zone rectangles
zone rotation
zone style overrides, если они привязаны к зоне
```

Что не трогать:

```text
markers
devices
rooms
images
security
attention
PIN
```

### Подтверждение

Обе функции опасные для layout:

```text
доступны только admin
требуют подтверждение
желательно hold 3 seconds
перед выполнением создавать backup layout.json
```

Рекомендуемые подписи:

```text
Очистить маркеры
Удерживайте 3 секунды

Очистить зоны
Удерживайте 3 секунды
```

### Backup

Перед очисткой создать:

```text
/data/backups/layout-before-clear-markers-YYYYMMDD-HHMMSS.json
/data/backups/layout-before-clear-zones-YYYYMMDD-HHMMSS.json
```

### API

```text
POST /api/layout/clear-markers
POST /api/layout/clear-zones
```

### Критерии готовности

```text
- можно очистить только маркеры;
- можно очистить только зоны;
- backup создаётся;
- devices не удаляются;
- rooms/images/security/attention не затрагиваются;
- после очистки UI обновляется.
```

## C. Профили проекта

### Цель

Добавить возможность иметь до 3 параллельных профилей проекта.

Профиль — это отдельная конфигурация отображения дома/плана:

```text
свои картинки
свой layout
свои zones
свои rooms settings
своё размещение устройств
свои пользовательские UI-настройки отображения
```

Но настройки безопасности должны быть общими.

### Зачем это нужно

Примеры:

```text
Профиль 1: Основной 3D-план
Профиль 2: Альтернативный план / другой этаж / другой стиль картинок
Профиль 3: Тестовый профиль для экспериментов
```

или:

```text
Профиль 1: Квартира
Профиль 2: Дом
Профиль 3: Дача
```

### Ограничение

Максимум:

```text
3 профиля
```

---

## C1. Что у каждого профиля своё

Для каждого профиля отдельно:

```text
layout.json
rooms.json
images/
images_meta.json
source_config.json, если нужно
ui_state defaults, если нужно
zone settings
marker placement
room image assignment
overview image
```

Рекомендуемая структура:

```text
/data/profiles/
  profile-1/
    layout.json
    rooms.json
    images/
      overview/
      rooms/
      originals/
      images_meta.json
    source_config.json

  profile-2/
    layout.json
    rooms.json
    images/
    source_config.json

  profile-3/
    layout.json
    rooms.json
    images/
    source_config.json
```

Общий файл профилей:

```text
/data/profiles.json
```

Пример:

```json
{
  "version": 1,
  "activeProfileId": "profile-1",
  "profiles": [
    {
      "id": "profile-1",
      "name": "Основной",
      "createdAt": "2026-05-07T00:00:00.000Z",
      "updatedAt": "2026-05-07T00:00:00.000Z"
    }
  ]
}
```

---

## C2. Что общее для всех профилей

Не дублировать по профилям:

```text
security_rules.json
PIN
roles
dangerous overrides
attention_rules.json, если мониторинг должен быть глобальным
command_log.json
основной addon_config security section
```

### Security/PIN

Security/PIN должны быть общими:

```text
PIN один на всё приложение
roles общие
dangerous rules общие
admin/control/viewer общие
```

### Attention Monitor

Рекомендация:

```text
attention_rules.json оставить глобальным
```

Почему:

```text
alert должен работать независимо от текущей комнаты и профиля;
если датчик важный, он важный для всей панели;
kiosk “!” должен быть глобальным.
```

---

## C3. Управление профилями

Доступно только:

```text
admin mode
```

Раздел:

```text
Настройки → Профили
```

Кнопки:

```text
Создать профиль
Дублировать профиль
Удалить профиль
Переключить профиль
Переименовать профиль / задать отображаемое имя
```

### Создать профиль

Создаёт новый пустой профиль:

```text
fallback overview
нет custom room images
layout пустой или минимальный
rooms строятся из Lovelace/HA
zones пустые
markers не размещены
```

Если уже есть 3 профиля:

```text
кнопка “Создать профиль” disabled
показать текст: Максимум 3 профиля
```

### Дублировать профиль

Копирует текущий профиль:

```text
layout
zones
images
rooms settings
images_meta
source_config, если profile-specific
```

Не копирует как отдельные:

```text
PIN
security
attention
command log
```

После дублирования:

```text
создать новый profile_id
название: “Копия <имя профиля>”
сразу переключиться или спросить
```

### Удалить профиль

Правила:

```text
удалять можно только не последний профиль
нельзя удалить active profile без подтверждения выбора другого
удаление требует подтверждение / hold 3 seconds
перед удалением создать backup профиля
```

Если профиль активный:

```text
попросить выбрать другой профиль
или переключить на первый доступный после удаления
```

### Переключить профиль

При переключении:

```text
сохранить текущий UI state
загрузить layout/images/rooms нового профиля
перерисовать UI
HA states не сбрасывать
attention/security остаются общими
```

## C5. Миграция старых путей

Нужно решить: оставить совместимость со старым `/data/layout.json` или перейти на `/data/profiles/profile-1/layout.json`.

### Рекомендованный путь миграции

При первом запуске версии с профилями:

```text
1. Если /data/profiles.json нет:
   создать profile-1
2. Перенести существующие:
   /data/layout.json → /data/profiles/profile-1/layout.json
   /data/rooms.json → /data/profiles/profile-1/rooms.json
   /data/images → /data/profiles/profile-1/images
3. Старые файлы оставить как backup или compatibility copy.
```

### Compatibility

Чтобы не сломать существующих пользователей:

```text
чтение сначала activeProfile paths
если нет — fallback на старые /data/*.json
после успешного сохранения писать уже в profile path
```

```text

```

## C8. Критерии готовности профилей

```text
- можно создать до 3 профилей;
- нельзя создать 4-й профиль;
- можно дублировать текущий профиль;
- можно удалить профиль, но не последний без замены;
- можно переключить активный профиль;
- каждый профиль имеет свои картинки/layout/zones;
- security/PIN остаются общими;
- attention остаётся глобальным;
- существующая установка мигрирует в profile-1 без потерь;
- backup создаётся перед миграцией/удалением.
```

## E. Обновлённые важные правила

```text
1. Пользовательские картинки можно загрузить, заменить и сбросить.
2. Очистка маркеров и очистка зон — разные операции.
3. Очистка layout всегда через backup и подтверждение.
4. Можно иметь до 3 профилей.
5. Профили имеют свои layout/images/zones/rooms settings.
6. Security/PIN/dangerous rules общие для всех профилей.
7. Attention Monitor рекомендуется оставить глобальным.
8. Переключение профилей доступно только admin.
9. Дублирование профиля нужно для безопасных экспериментов.
10. Миграция существующего проекта должна создать profile-1 без потерь.
```
